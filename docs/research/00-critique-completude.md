The research is strong on six fronts: notch-window mechanics (panel config, geometry, multi-screen), Claude Code's event/data surfaces (hooks incl. the native http hook type, stream-json, transcript schema, resume/concurrency limits), the ASCII aesthetic (fonts/licensing, shaders, theming), macOS packaging/distribution, and the competitive landscape including a deep Vibe Island teardown. However, it stops at describing WHAT the reference products do for four implementation-critical capabilities without researching HOW to build them, and each would materially change the plan. (1) Click-to-jump back to the exact terminal tab/pane or Cursor window is a headline feature with zero implementation research — it determines whether the app needs the Apple Events TCC entitlement, process-tree/TTY resolution at hook time, and possibly a bundled VS Code/Cursor extension (a whole extra deliverable). (2) Session liveness/ghost-session detection is the category's #1 reliability complaint, yet there is no research on PID discovery from hooks, kqueue process-exit monitoring, or fallback process/transcript scanning — this defines the core session state machine. (3) The exact hook mechanics for answering questions and approving plan-mode (ExitPlanMode/AskUserQuestion schemas, whether hook decisions can inject choices/free text, races with the terminal's own TUI prompt) define what 'interact without switching to the IDE' can actually deliver — the generic PreToolUse decision JSON is documented but these specific flows are not. (4) Usage/quota display (a signature Vibe Island feature and a known accuracy differentiator) has three architecturally different implementation routes (statusline bridge vs OAuth usage endpoint via keychain token vs JSONL cost parsing) that were named but not researched; the choice affects onboarding, credentials handling, and ToS risk. Everything else missing (sound packs, hook-install merge strategy, computer-use click passthrough) is resolvable at implementation time without changing the plan's shape.
