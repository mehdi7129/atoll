# gui-answering-mechanics

## Summary
An external GUI can answer Claude Code's interactive moments for sessions it does not own via blocking hooks in ~/.claude/settings.json: a `PermissionRequest` hook (matcher "*", "timeout": 86400) receives every permission dialog's tool_name/tool_input on stdin, blocks while forwarding to the GUI over a Unix socket, and answers by printing `{"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{...}}}`. Plan approval works because ExitPlanMode goes through this same dialog: `decision:{"behavior":"allow"}` approves the plan (optionally with `updatedPermissions` like setMode acceptEdits), and `{"behavior":"deny","message":"<feedback>"}` blocks the tool so Claude stays in plan mode and revises using the message. AskUserQuestion also fires PermissionRequest; the hook answers it with `behavior:"allow"` plus `updatedInput` echoing the original `questions` array and adding an `answers` object mapping question text to the chosen label (arrays or ", "-joined for multiSelect, arbitrary text for free-form, optional `response` for a general freeform reply) — open-vibe-island ships exactly this, while vibe-notch punts questions to the terminal. Elicitation/ElicitationResult hooks are MCP-only: they carry `mcp_server_name`, `message`, `mode` (form/url), `requested_schema`, and respond with `action` accept/decline/cancel plus `content` form values. Critically, the terminal TUI renders its own permission prompt concurrently while the hook blocks (confirmed by issue #12176: the dialog is added to UI state immediately and the hook runs async; a hook decision filters the dialog out by toolUseID), so there is a real race — first answer wins, the CLI discards the loser, and both open-source apps cancel their GUI card when PostToolUse/Stop reveals the terminal answered first. There is no "defer" for PermissionRequest — handing control back to the terminal is done by exiting 0 with no stdout (silence never approves); PreToolUse's `permissionDecision:"defer"` only works in headless `-p` mode and is ignored with a warning in interactive sessions. The official VS Code/Cursor extension does NOT fire PermissionRequest hooks (issue #16237, silently ignored), so this whole mechanism only works for CLI sessions in a real terminal (including Cursor's integrated terminal); Cursor's native agent uses an entirely different hook protocol (beforeShellExecution etc.). Interactions not reachable via hooks: injecting new prompts into idle sessions, interrupting a running turn, extension-panel sessions, and startup/trust/login dialogs — the two apps fall back to tmux send-keys / keystroke injection for those.

## Key findings
- ExitPlanMode tool_input schema (official hooks reference): {plan: string (markdown, INJECTED by the CLI from the plan file on disk — the model's literal tool_input is typically empty), planFilePath: string (injected), allowedPrompts: deprecated/ignored >=2.1.205}. In PostToolUse, tool_response.plan holds the approved plan.
- PermissionRequest hook input: common fields + tool_name, tool_input, permission_suggestions (array of 'always allow' options: {type:addRules, rules:[{toolName, ruleContent?}], behavior, destination}); NO tool_use_id (vibe-notch caches tool_use_id from the preceding PreToolUse keyed by session+tool+input to correlate).
- PermissionRequest decision schema: {behavior: allow|deny; updatedInput (allow only, replaces ENTIRE input object, re-evaluated against deny/ask rules); updatedPermissions (allow only, same entry objects as permission_suggestions incl. setMode default|auto|acceptEdits|dontAsk|bypassPermissions|plan, destination session|localSettings|projectSettings|userSettings); message (deny only, shown to Claude); interrupt (deny only, true stops Claude's turn)}. Wrapped as {"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{...}}}; a hook allow does NOT override matching deny/ask rules.
- Plan approval verified: PermissionRequest hook with behavior:allow = user pressing yes (plan proceeds, plan mode exits); behavior:deny + message = plan rejected with feedback, ExitPlanMode never executes, Claude remains in plan mode and revises; add updatedPermissions [{type:setMode, mode:acceptEdits, destination:session}] to emulate the 'yes, and auto-accept edits' option; echoing a received permission_suggestion as updatedPermissions equals clicking that 'always allow' choice.
- AskUserQuestion tool_input: {questions:[{question, header (<=12 chars), options:[{label, description, preview?}] (2-4), multiSelect:bool}] (1-4 questions)}. Answer via hook: behavior:allow + updatedInput {questions:<passthrough, required>, answers:{"<question text>":"<label>"}} — multiSelect = array of labels or ','-joined; free-form text = put the user's raw text as the value (an 'Other' option is added client-side by the CLI and should be mirrored, per open-vibe-island's comment); optional top-level 'response' string = general freeform reply replacing per-question answers ('The user responded: ...'). The 'answers' field is documented in the official hooks reference: 'Claude doesn't set this field; supply it via updatedInput to answer programmatically'.
- AskUserQuestion triggers BOTH PreToolUse and PermissionRequest hooks (issue #15400); AskUserQuestion always prompts even when an allow rule matches (it's a requiresUserInteraction tool); in headless -p mode only PreToolUse allow+updatedInput(answers) works (allow alone is insufficient — official docs section 'Defer a tool call for later' + line: 'AskUserQuestion and ExitPlanMode require user interaction and normally block in non-interactive mode').
- Elicitation hook (MCP servers ONLY, matcher = server name): input {mcp_server_name, message, mode:'form'|'url', requested_schema (JSON schema of form fields), url?, elicitation_id?}; output {"hookSpecificOutput":{"hookEventName":"Elicitation","action":"accept|decline|cancel","content":{field:value}}} — accepting with content skips the dialog entirely. ElicitationResult fires after the user responds and can override {action, content} before it reaches the server; exit code 2 = decline/block. These events do NOT apply to AskUserQuestion.
- RACE (part c, confirmed by issue #12176 + both apps' code): the terminal permission dialog is added to UI state immediately and rendered while the PermissionRequest hook runs asynchronously — the TUI prompt and the blocking hook are live simultaneously. Hook decision arriving later filters the dialog from state (dismissed, session continues); user answering the terminal first wins and the late hook output is discarded. Both apps detect terminal-side resolution via PostToolUse (tool_use_id) / Stop and cancel their GUI card: vibe-notch HookSocketServer.cancelPendingPermission(toolUseId:) closes the client socket so the python hook exits silently; open-vibe-island clearStaleClaudeInteractionIfNeeded emits 'Approval was handled outside Open Island'.
- No 'defer' exists for PermissionRequest. Handing control back to the terminal prompt = exit 0 with NO stdout JSON ('The hook can deny the call, but staying silent doesn't approve it' — normal permission flow continues with the already-visible dialog). vibe-notch's 'ask' decision does exactly this (sys.exit(0)). permissionDecision:'defer' is PreToolUse-only, honored ONLY in non-interactive -p mode (process exits with stop_reason:'tool_deferred' + deferred_tool_use {id,name,input}; resume with --resume; single-tool-call turns only); in interactive sessions defer logs a warning and is ignored. Hook timeout expiry = canceled, treated as no decision, dialog remains (fail-open). PreToolUse multi-hook precedence: deny > defer > ask > allow.
- TUI display when a hook decides: the permission dialog is dismissed (or never rendered if the hook returns in <~1.5s) and the transcript continues as if the user chose; deny messages surface as feedback to Claude; no dedicated 'decided by hook' banner is documented for PermissionRequest (PreToolUse 'ask' prompts do get a source label like [User]/[Project]/[Plugin]/[Local]). As of v2.1.139 hook processes run without a controlling terminal (cannot write /dev/tty). Notification hook fires with notification_type:'permission_prompt' when the prompt is shown (vibe-notch skips it in favor of PermissionRequest).
- (d) The official Claude Code VS Code/Cursor extension does NOT trigger PermissionRequest hooks — silently ignored (issue #16237, Jan 2026, open feature request); the same settings.json works fine for CLI sessions in the integrated terminal. Agent-Teams subagent permission requests also don't fire PermissionRequest hooks (issue #23983). Cursor's NATIVE agent has a separate hook protocol (beforeSubmitPrompt/beforeShellExecution/beforeMCPExecution/beforeReadFile/afterFileEdit/stop, response {continue, permission:allow|deny, agentMessage}) handled by open-vibe-island's CursorHooks.swift — not Claude Code hooks.
- open-vibe-island (github.com/Octane0411/open-vibe-island) concrete JSON: installs 14 events via ClaudeHookInstaller (PermissionRequest matcher '*' timeout 86400, managedTimeout constant); client = OpenIslandHooks binary or scripts/open-island-hooks.py '--source claude' -> Unix socket /tmp/open-island-$UID.sock (env OPEN_ISLAND_SOCKET_PATH), python socket timeout 86400 for PermissionRequest / 45s else. Approve: {"continue":true,"suppressOutput":true,"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow","updatedInput":{...},"updatedPermissions":[...]}}}. Deny: decision {"behavior":"deny","message":"Permission denied in Open Island.","interrupt":false}. Question answer (BridgeServer.mergedClaudeQuestionInput): decision allow + updatedInput = original tool_input with added answers:{questionText:answerString} and optional annotations:{key:{preview,notes}}. Also emits PreToolUse directives {permissionDecision, permissionDecisionReason, updatedInput, additionalContext}. Key files: Sources/OpenIslandCore/ClaudeHooks.swift (encoder ~line 598), BridgeServer.swift (resolvePendingClaudeQuestion ~2469, race handling ~1805), ClaudeHookInstaller.swift, docs/hooks.md.
- vibe-notch (github.com/farouqaldori/vibe-notch, app name ClaudeIsland) concrete JSON: HookInstaller writes ~/.claude/settings.json entries running 'python3 ~/.claude/hooks/claude-island-state.py' for baseline events + version-gated ones (PostToolUseFailure >=2.0.0, SubagentStart >=2.0.43, PostCompact >=2.1.76, StopFailure >=2.1.78, PermissionDenied >=2.1.88 — detected via 'claude --version'); PermissionRequest gets matcher '*' + timeout 86400 while the python socket read timeout is 300s. Approve: {"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow"}}}; Deny: {"behavior":"deny","message":"Denied by user via ClaudeIsland"}; 'ask'/no app response -> sys.exit(0) (terminal dialog handles it). It does NOT answer AskUserQuestion or inject plan feedback via hooks — AskUserQuestion is flagged isInteractiveTool and the UI shows 'Claude Code needs your input' + Go-to-Terminal (tmux/yabai); ExitPlanMode gets plain Allow/Deny buttons; plan markdown is read from the session transcript JSONL (toolUseResult {plan, filePath, isAgent}) for display. Socket /tmp/claude-island.sock, request/response only for status 'waiting_for_approval'.
- NOT achievable via hooks: (1) answering anything in extension-panel sessions (PermissionRequest hooks don't fire there); (2) injecting a new user prompt / follow-up into an idle session or steering mid-turn (no hook starts a turn — both apps use tmux send-keys or CGEvent keystroke injection for chat); (3) interrupting a running turn (Esc) — deny+interrupt only works at a permission checkpoint; (4) an explicit defer-to-terminal decision on PermissionRequest (only silent exit 0); (5) startup/trust-folder//login/model-picker dialogs; (6) MCP tools marked _meta['anthropic/requiresUserInteraction'] (>=2.1.199) — hook allow cannot skip their prompt; connector tools org-set to 'ask' also always prompt; (7) editing the plan text itself before approval (plan is injected from a file Claude wrote; updatedInput.plan changes what executes, not the authored plan) — unverified/undocumented; (8) subagent (Agent Teams) permission dialogs (issue #23983).

## Recommendations
- Register PermissionRequest with matcher '*' and per-hook "timeout": 86400 (both reference apps do); keep every other event at default timeout and make the hook client fail open (exit 0, no stdout) whenever the GUI socket is unreachable so the terminal is never blocked.
- Cache tool_use_id from PreToolUse keyed by session_id+tool_name+serialized tool_input (vibe-notch and open-vibe-island both do this) because PermissionRequest input lacks tool_use_id; use it to correlate and to cancel the GUI card when PostToolUse/Stop shows the user answered in the terminal — treat the terminal and GUI as racing surfaces where first answer wins.
- For plan approval: render tool_input.plan (markdown) from the ExitPlanMode PermissionRequest payload; respond allow (optionally updatedPermissions [{type:setMode, mode:acceptEdits, destination:session}] for an 'approve + auto-accept edits' button, or echo a permission_suggestions entry for 'always allow'); respond deny with the user's typed feedback as message to send Claude back to revise; never set interrupt:true unless the user explicitly stops the session.
- For questions: detect tool_name == 'AskUserQuestion' in the PermissionRequest hook and answer with behavior:allow + updatedInput {questions: <original array passthrough>, answers: {questionText: label-or-freetext}}; add a GUI 'Other' free-text option per question; use ', '-joined labels for multiSelect; fall back to exit 0 (terminal) if the user dismisses the GUI card.
- Detect the host: if the session runs under the VS Code/Cursor extension panel, PermissionRequest hooks never fire (issue #16237) — surface those sessions as read-only/'answer in editor' in the GUI; sessions in any real terminal (incl. Cursor's integrated terminal) work fully. For prompt injection / interrupts, implement a terminal-side channel (tmux send-keys or macOS keystroke injection) as both reference apps do, since hooks cannot do it.
- Support Elicitation/ElicitationResult (matcher = MCP server name) as a separate GUI form flow using requested_schema; do not conflate it with AskUserQuestion.
- Re-verify the double-answer discard behavior and hook-decision TUI dismissal on the currently installed CLI build (claude --version), since issue #12176 showed the dialog/hook race timing has shifted across 2.x releases; gate optional events by version like vibe-notch does (PermissionDenied needs >=2.1.88) because unknown hook keys can break older settings.json parsing.
