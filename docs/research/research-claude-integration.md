# claude-integration

## Summary
A native macOS app has four integration surfaces with Claude Code (verified locally against CLI v2.1.214 and current code.claude.com docs). (1) Headless mode: `claude -p` with `--output-format stream-json --verbose --include-partial-messages` emits NDJSON events (system/init, stream_event wrapping raw Anthropic SSE deltas, assistant, user, rate_limit_event, result with cost/usage) — I captured a real stream and documented every field; `--input-format stream-json` keeps one process alive for true multi-turn interactive sessions driven over stdin. (2) Hooks: configured in ~/.claude/settings.json under a "hooks" key; ~30 events exist in 2026 (core: PreToolUse, PostToolUse, Notification, Stop, SubagentStop, UserPromptSubmit, SessionStart, SessionEnd) and each receives a documented JSON payload on stdin including session_id, transcript_path, cwd, tool_name/tool_input; there is now a native `"type":"http"` hook that POSTs the payload to a localhost URL, so a GUI app just runs an HTTP server — no curl needed (though a command hook piping stdin to curl also works). Hooks execute inside the claude CLI process itself, so global hooks fire identically for sessions launched from Cursor's integrated terminal; the known Cursor bug about .claude/settings.json hooks concerned Cursor's own native agent, not the CLI, and was fixed. (3) Transcripts live at ~/.claude/projects/<cwd-with-non-alphanumerics-replaced-by-dashes>/<session-uuid>.jsonl, appended live, with a sidecar <session-uuid>/ dir holding subagents/agent-*.jsonl, workflows/, and tool-results/; I extracted the actual line schema (types: user, assistant, system, attachment, file-history-snapshot, mode, permission-mode, last-prompt, ai-title; envelope: parentUuid/uuid/timestamp/sessionId/cwd/gitBranch/version), but Anthropic explicitly documents the format as internal and version-unstable. (4) The Agent SDK (npm @anthropic-ai/claude-agent-sdk, which bundles the CLI binary per-platform; pip claude-agent-sdk) offers query()/ClaudeSDKClient with in-process canUseTool permission callbacks, hooks-as-callbacks, interrupt(), and listSessions()/getSessionMessages() — a Swift app would run it in a Node sidecar, but note third-party products must use API-key auth, not claude.ai login. (5) Sending a new prompt to an existing session is officially `claude -p --resume <session-id> "prompt"` from the same cwd (resume reuses the session ID by default; --fork-session creates a new one); concurrent use of one session by two clients is not locked — docs state messages from both interleave into one transcript — so fork or use a single owning process with stream-json stdin instead.

## Key findings
- HEADLESS FLAGS (verified on claude 2.1.214): `claude -p/--print [prompt]`; `--output-format text|json|stream-json` and `--input-format text|stream-json` are print-mode only; stream-json output requires `--verbose`; `--include-partial-messages` adds stream_event lines (requires -p + stream-json); `--include-hook-events` adds hook lifecycle events for ALL hooks (SessionStart/Setup hook events are always emitted regardless); `--replay-user-messages` echoes stdin user messages back on stdout for acks (requires stream-json in+out).
- STREAM-JSON EVENT TYPES (captured live): {type:'system',subtype:'init'} first (fields: cwd, session_id, tools[], mcp_servers[{name,status}], model, permissionMode, slash_commands[], apiKeySource, claude_code_version, output_style, agents[], skills[], plugins[], capabilities[] e.g. ['interrupt_receipt_v1','msg_lifecycle_v1'], memory_paths, uuid); {type:'system',subtype:'status'|'hook_started'|'hook_response'|'api_retry'|'plugin_install'}; {type:'stream_event',event:<raw Anthropic SSE: message_start|content_block_start|content_block_delta{delta:{type:'text_delta',text}}|content_block_stop|message_delta|message_stop>, session_id, parent_tool_use_id, uuid, ttft_ms}; {type:'assistant'|'user', message:<full API Message with content[], usage>, parent_tool_use_id, session_id, uuid, timestamp}; {type:'rate_limit_event',rate_limit_info:{status,resetsAt,rateLimitType,overageStatus}}; final {type:'result',subtype:'success'|error, is_error, duration_ms, duration_api_ms, ttft_ms, num_turns, result, stop_reason, session_id, total_cost_usd, usage, modelUsage{<model>:{inputTokens,outputTokens,costUSD,contextWindow}}, permission_denials[], terminal_reason, uuid}.
- SUBAGENT ATTRIBUTION in stream: messages carry parent_tool_use_id (null for main conversation, tool-call id for subagent); by default only subagent tool_use/tool_result are forwarded — pass `--forward-subagent-text` (v2.1.211+) to also get subagent text/thinking.
- PERMISSIONS: `--permission-mode` accepts acceptEdits|auto|bypassPermissions|manual|dontAsk|plan on 2.1.214 ('manual' is alias for 'default' since v2.1.200; docs still list 'default'). `--permission-prompt-tool <mcp_tool_name>` still exists but is HIDDEN from --help (verified: CLI accepts it, errors 'argument missing'); it routes non-interactive permission prompts to an MCP tool; since v2.1.206 Claude waits up to MCP_TIMEOUT (30s) for that server. `--allowedTools/--disallowedTools` use permission-rule syntax like Bash(git diff *).
- STREAM-JSON INPUT (multi-turn in ONE process — best for a GUI chat): run `claude -p --input-format stream-json --output-format stream-json --verbose` and write NDJSON user messages to stdin: {"type":"user","message":{"role":"user","content":"..."|[{type:'text',...},{type:'image',source:{type:'base64',media_type,data}}]},"parent_tool_use_id":null}. Supports queued messages, images, and interruption (the SDK's interrupt() rides this control channel; init capabilities lists interrupt_receipt_v1).
- OTHER KEY -p FLAGS: `--session-id <uuid>` (pre-choose the UUID so the app can watch the transcript file immediately), `--no-session-persistence`, `--max-budget-usd <amt>`, `--fallback-model a,b`, `--json-schema '<schema>'` (result gains structured_output field; invalid schema errors since v2.1.205), `--bare` (fast minimal mode — but it SKIPS hooks and OAuth/keychain, auth only via ANTHROPIC_API_KEY, so do NOT use --bare if relying on the user's subscription login or on hooks), `--bg` + `claude agents` (background agents), `--remote-control`. Piped stdin capped at 10MB (v2.1.128+). SIGTERM aborts turn, runs SessionEnd hooks, exits 143. Background bash shells killed ~5s after final result.
- HOOKS CONFIG: settings files = ~/.claude/settings.json (all projects), .claude/settings.json, .claude/settings.local.json, plugin hooks/hooks.json, skill/agent frontmatter. Shape: {"hooks":{"PreToolUse":[{"matcher":"Bash|Edit","hooks":[{"type":"command","command":"...","timeout":30,"async":true,"asyncRewake":false,"if":"Bash(rm *)","statusMessage":"..."}]}]}}. Matchers: exact, pipe/comma list, unanchored regex, * or "" = all. disableAllHooks:true kills all. Handler types: command, http, mcp_tool, prompt, agent.
- HOOK EVENTS (2026 full list): SessionStart, Setup, UserPromptSubmit, UserPromptExpansion, PreToolUse, PermissionRequest, PermissionDenied, PostToolUse, PostToolUseFailure, PostToolBatch, Notification, MessageDisplay, SubagentStart, SubagentStop, TaskCreated, TaskCompleted, Stop, StopFailure, TeammateIdle, InstructionsLoaded, ConfigChange, CwdChanged, FileChanged, WorktreeCreate, WorktreeRemove, PreCompact, PostCompact, Elicitation, ElicitationResult, SessionEnd.
- HOOK STDIN PAYLOAD (JSON): common fields session_id, prompt_id, transcript_path, cwd, permission_mode, effort{level}, hook_event_name (+agent_id/agent_type inside subagents). Event-specific: PreToolUse/PermissionRequest add tool_name+tool_input; PostToolUse adds tool_response; UserPromptSubmit adds prompt; Notification adds type+message (matcher values: permission_prompt, idle_prompt, auth_success, elicitation_dialog); SessionStart adds source(startup|resume|clear|compact)+model+session_title; Stop/SubagentStop add last_assistant_message; SessionEnd matcher = end reason. Exit codes: 0=success (stdout JSON parsed), 2=blocking error (stderr fed back to Claude), other=non-blocking. Decision JSON: PreToolUse → {hookSpecificOutput:{hookEventName:'PreToolUse',permissionDecision:'allow|deny|ask|defer',permissionDecisionReason,updatedInput,additionalContext}}; Stop/UserPromptSubmit/PostToolUse → top-level {decision:'block',reason}; universal {continue:false,stopReason,systemMessage,suppressOutput}.
- GUI EVENT BRIDGE: native HTTP hook type — {"type":"http","url":"http://localhost:8080/hooks/pre-tool-use","timeout":600,"headers":{...}} POSTs the same JSON payload to the app's local server (2xx empty = ok, 2xx JSON = parsed as decision, non-2xx/timeout = non-blocking error). Command-hook alternative: {"type":"command","command":"curl -s -m 2 -X POST http://127.0.0.1:PORT/claude-event -H 'Content-Type: application/json' --data-binary @-","async":true} (stdin pipes straight to curl; async:true avoids adding latency; for Stop/PreToolUse leave sync only if the app must be able to block/decide).
- CURSOR CONFIRMATION: hooks are executed by the claude CLI process itself, and ~/.claude/settings.json is scoped 'all projects', so global hooks DO fire for claude sessions started inside Cursor's integrated terminal (any terminal host). The forum-reported failure (forum.cursor.com/t/claude-hooks-dont-work/153614) was about Cursor's OWN native agent not loading .claude/settings.json hooks — fixed in a 2026 Cursor update — and never affected the CLI. Caveats: hooks do NOT run with --bare, --safe-mode, or disableAllHooks:true.
- TRANSCRIPTS: ~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl where encoded-cwd = absolute path with every non-alphanumeric char replaced by '-' (/Users/me/proj → -Users-me-proj); relocatable via CLAUDE_CONFIG_DIR; 30-day retention via cleanupPeriodDays setting. Files are appended live — FSEvents-watchable. Sidecar per session: <session-uuid>/ dir with subagents/agent-<hex>.jsonl + .meta.json, workflows/*.json, tool-results/*.txt; per-project memory/ dir alongside.
- JSONL LINE SCHEMA (extracted from real v2.1.x files): line types = user, assistant, system, attachment, file-history-snapshot, mode, permission-mode, last-prompt, ai-title, summary, progress. Envelope fields on message lines: parentUuid (conversation-tree link, null at root), isSidechain, promptId, uuid, timestamp, sessionId, session_id, userType, entrypoint, cwd, version, gitBranch, isMeta. assistant lines: message = full API Message {model,id,type,role,content[],stop_reason,usage} + requestId + effort; tool_use blocks = {type,id,name,input,caller}; user tool_result lines add toolUseResult (rich structured result) + sourceToolAssistantUUID. ai-title/last-prompt lines carry the picker metadata. WARNING: Anthropic documents this format as internal and version-unstable — parse defensively; sanctioned interfaces are hooks' transcript_path, /export, `claude -p --resume`, and SDK listSessions()/getSessionMessages().
- AGENT SDK: TypeScript npm @anthropic-ai/claude-agent-sdk — bundles native CLI binaries as optional deps (@anthropic-ai/claude-agent-sdk-darwin-arm64 etc.; SDK 0.3.191 bundles CLI 2.1.191; override with pathToClaudeCodeExecutable). Python pip claude-agent-sdk (Python ≥3.10; ClaudeSDKClient for stateful multi-turn). query({prompt: string|AsyncIterable<SDKUserMessage>, options}) returns AsyncGenerator + Query methods: interrupt(), setPermissionMode(), setModel(), applyFlagSettings(), streamInput(), stopTask(), close(), mcpServerStatus(). Key Options: canUseTool(request,{signal})→PermissionResult (in-process permission UI — the SDK replacement for --permission-prompt-tool), hooks (callbacks, same event names), resume, forkSession, continue, persistSession (TS only; false = memory-only), sessionStore (cross-host), includePartialMessages, maxTurns, maxBudgetUsd, settingSources, env, cwd, abortController. Session utilities: listSessions(), getSessionMessages(), getSessionInfo(), renameSession(), tagSession(). NOTE: TS 'V2 session API' (createSession/send/stream) was REMOVED in 0.3.142. AUTH: third-party products must use ANTHROPIC_API_KEY (or Bedrock/Vertex/Foundry); claude.ai subscription login is not permitted for third-party SDK products, and branding may not say 'Claude Code'.
- NEW PROMPT TO EXISTING SESSION: one-shot documented pattern `claude -p --resume <session-id> --output-format json "summarize what we changed" | jq -r '.result'` — MUST run from the original cwd (session-ID lookup is scoped to the project directory + its git worktrees; otherwise 'No conversation found with session ID'). --resume REUSES the same session ID by default in current versions; add --fork-session (CLI) / forkSession:true (SDK) to branch to a new ID leaving the original intact. -p sessions don't appear in the /resume picker but resume fine by ID. Resume restores conversation/model/permission-mode (never plan or bypassPermissions) but NOT --mcp-config/--settings/--plugin-dir/--add-dir flags — repass them.
- CONCURRENCY LIMITATION (documented verbatim in sessions doc): 'If you resume the same session in two terminals without forking, messages from both interleave into one transcript.' There is no lock and no supported IPC to inject a prompt into another client's RUNNING interactive session. For a GUI: either (a) observe foreign sessions read-only via hooks + transcript tailing, (b) fork before prompting a session that may be active elsewhere, or (c) own the session exclusively in a persistent stream-json stdin process.

## Recommendations
- For the live-events feed into the macOS app (menu bar / Dynamic Island style UI): install global hooks in ~/.claude/settings.json for SessionStart, UserPromptSubmit, PreToolUse, PostToolUse, Notification, Stop, SubagentStop, SessionEnd using the native {"type":"http","url":"http://127.0.0.1:<port>/claude-event"} handler (or command+curl with async:true as fallback); correlate events by session_id and use transcript_path+cwd from the payload to attach richer context. This covers every session on the machine, including ones started in Cursor's or any other terminal.
- Keep event hooks async/non-blocking except where the app should act as a permission UI — there use a synchronous PreToolUse or PermissionRequest hook (HTTP type) and answer with hookSpecificOutput.permissionDecision allow/deny/ask within the timeout.
- To render full conversation history and live-tail activity, watch ~/.claude/projects/ with FSEvents and parse the JSONL defensively (unknown line types and fields must be ignored; the schema is officially internal and changes between releases); prefer hook payloads for real-time triggers and JSONL only for display.
- To drive Claude from the app, spawn /usr/bin/env claude via Foundation Process with `-p --input-format stream-json --output-format stream-json --verbose --include-partial-messages` and keep it alive for multi-turn chat over stdin; pre-set --session-id (or capture system/init.session_id) so the app can immediately find the transcript; do NOT use --bare (it disables hooks and subscription auth).
- For follow-up prompts to past sessions use `claude -p --resume <id>` launched with the session's original cwd; if the session might be open in a terminal simultaneously, add --fork-session to avoid interleaving two writers into one transcript.
- Consider a Node sidecar embedding @anthropic-ai/claude-agent-sdk only if you need in-process canUseTool permission callbacks, interrupt(), or listSessions()/getSessionMessages() convenience — but note the API-key-only auth policy for shipped third-party products; for a personal/local companion app that piggybacks on the user's own CLI login, spawning the installed claude binary is the safer and simpler backend.
- Feature-detect via the system/init capabilities array (e.g. interrupt_receipt_v1) rather than comparing version strings; handle system/api_retry and rate_limit_event lines to surface progress and quota state in the UI.
