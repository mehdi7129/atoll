# terminal-jump-back

## Summary
Click-to-jump from a notch app to the exact terminal tab/pane hosting a Claude Code session is a solved pattern: capture host identity at hook time (the hook runs as a descendant of the claude CLI, so it inherits both the session's controlling TTY and the terminal's environment variables), then use a per-terminal adapter to focus. On macOS, proc_pidinfo(PROC_PIDTBSDINFO) alone yields both pbi_ppid (for walking up to claude and the terminal-app ancestor) and e_tdev (controlling TTY, convertible via devname()) — verified in the local SDK's sys/proc_info.h. Per-tab/pane targeting by TTY works natively for Terminal.app and iTerm2 (AppleScript 'tty' property + select), for WezTerm (cli list --format json exposes tty_name; activate-pane --pane-id), kitty (kitten @ focus-window --match, using $KITTY_WINDOW_ID captured at hook time), and tmux (list-panes -a matched on pane_tty, then select-window/select-pane, plus recursive resolution of the client_tty to focus the outer terminal). Ghostty has AppleScript (1.3.0+, windows/tabs/terminals with select/focus) but no tty/pid property until the planned 1.4.0 (issue #11592), so today it needs a SessionStart-time probe of the focused terminal's id or cwd/title matching. Warp and Alacritty support only app-level activation (Warp's warp:// scheme only creates new tabs; WARP_SESSION_ID + warp://session/<id> was requested in issue #8611 and may be landing — feature-detect at runtime). For VS Code/Cursor, a bundled helper extension installed by folder-copy into ~/.vscode/extensions and ~/.cursor/extensions (or via code/cursor --install-extension) registers a vscode://-or-cursor:// URI handler, matches window.terminals by awaiting terminal.processId and checking ancestry against the claude PID, then calls terminal.show(). Sessions launched from the official Claude Code extension panel run in SDK mode (CLAUDE_CODE_ENTRYPOINT="sdk-ts", stdio pipes, no tty) — hooks still fire and the ancestor chain identifies the IDE, but focus is window-level only ('code -r <cwd>' / 'cursor -r <cwd>'), optionally plus executeCommand('claude-vscode.focus') from the helper extension; the official extension registers no onUri handler (verified in the locally installed anthropic.claude-code 2.1.214 manifest). Apple Events sending requires the com.apple.security.automation.apple-events entitlement + NSAppleEventsUsageDescription, produces one TCC prompt per (your app, target terminal) pair, fails silently with -1743 after denial (reset via tccutil reset AppleEvents <your-bundle-id>), can be pre-flighted with AEDeterminePermissionToAutomateTarget (target must be running), and all AppleScript must be executed by the main app — not the hook helper — so TCC attributes correctly. Accessibility (separate TCC service) is only a window-raise fallback; GPU-rendered UIs like Warp/Alacritty expose no usable per-tab AX tree.

## Key findings
- (a) PROCESS RESOLUTION — one API suffices: proc_pidinfo(pid, PROC_PIDTBSDINFO, 0, &info, MemoryLayout<proc_bsdinfo>.size) returns struct proc_bsdinfo containing pbi_ppid (parent pid), e_tdev (controlling tty dev_t) and e_tpgid — verified locally at $(xcrun --show-sdk-path)/usr/include/sys/proc_info.h lines 64/77/78. Convert tty: String(cString: devname(dev_t(info.e_tdev), mode_t(S_IFCHR))) -> "ttys012" -> prepend /dev/. e_tdev == UInt32.max (NODEV) means no controlling tty (IDE SDK mode). proc_pidpath(pid, &buf, UInt32(PROC_PIDPATHINFO_MAXSIZE)) and proc_name(pid,...) give executable path/name. Fallback: sysctl mib [CTL_KERN, KERN_PROC, KERN_PROC_PID, pid] -> kinfo_proc.kp_eproc.e_tdev. sysctl KERN_PROCARGS2 can read another same-user process's argv+env (useful for polling-based discovery outside hooks).
- (a) The hook command inherits claude's controlling TTY and full environment even though its stdin is the JSON pipe — so the helper can read its OWN e_tdev and getenv() instead of probing claude. (open-vibe-island instead shells out to `ps -p $PPID -o tty=` because /usr/bin/tty checks stdin — unnecessary in a native helper.) Walk pbi_ppid upward: first ancestor named "claude" (or "node" running claude) = claude PID; keep walking to find the terminal-app ancestor; resolve ancestor pids to bundle IDs with NSRunningApplication(processIdentifier:)?.bundleIdentifier in the MAIN app (helper just ships the raw pid chain).
- (a) tmux breaks the ancestor walk: panes are children of the tmux server (parent = launchd), never of the terminal app. Detect via TMUX env ("socketpath,pid,session") and TMUX_PANE ("%N"). Resolve outer terminal: `tmux list-clients -F '#{client_tty} #{client_session}'` -> client_tty is the OUTER terminal's /dev/ttysNNN -> recursively focus that tty in the host terminal adapter, then focus the pane inside tmux.
- (a) Env vars to snapshot at hook time (inherited from claude): TERM_PROGRAM (values: Apple_Terminal, iTerm.app, ghostty, WezTerm, vscode, tmux) + TERM_PROGRAM_VERSION, ITERM_SESSION_ID ("w0t0p0:UUID"), KITTY_WINDOW_ID / KITTY_PID / KITTY_LISTEN_ON, WEZTERM_PANE (numeric pane id) / WEZTERM_UNIX_SOCKET, TMUX / TMUX_PANE, GHOSTTY_RESOURCES_DIR, WARP_IS_LOCAL_SHELL_SESSION (+ possibly WARP_SESSION_ID in future builds), ALACRITTY_WINDOW_ID / ALACRITTY_SOCKET, VSCODE_INJECTION / VSCODE_GIT_ASKPASS_MAIN (path distinguishes VS Code vs Cursor), CURSOR_TRACE_ID, CLAUDE_CODE_ENTRYPOINT ("cli" = terminal, "sdk-ts" = IDE webview panel), __CFBundleIdentifier (bundle id of the app that spawned the tree — authoritative when present).
- (b) CAPABILITY MATRIX — Terminal.app (com.apple.Terminal): PANE-LEVEL(tab) by TTY. AppleScript: tell application "Terminal" / repeat with w in windows / repeat with t in tabs of w / if (tty of t as text) is "/dev/ttys004" then { set selected of t to true; set frontmost of w to true; activate } — verbatim pattern confirmed in open-vibe-island TerminalJumpService.swift.
- (b) iTerm2 (com.googlecode.iterm2): SPLIT-PANE-LEVEL by TTY or by session unique ID (== UUID part of ITERM_SESSION_ID). AppleScript: tell application "iTerm" / repeat with aWindow in windows / repeat with aTab in tabs of aWindow / repeat with aSession in sessions of aTab / if (tty of aSession) is "/dev/ttys004" (or (id of aSession as text) = storedID) then { select aWindow; select aTab; select aSession; activate }. AppleScript is frozen-but-working; Python API (session.async_activate / reveal) is the maintained alternative but requires user-enabled Python API permission.
- (b) Ghostty (com.mitchellh.ghostty): AppleScript since 1.3.0 (March 2026; on by default, disable via macos-applescript=false; source of truth = Ghostty.sdef). Hierarchy application->windows->tabs->terminals (terminals = splits); commands: `select` on tab, `focus` on terminal (raises window), input text, send key. Terminal properties are ONLY id/name/working directory — NO tty/pid until issue #11592 (milestone 1.4.0, PR #11922; stable was 1.3.1 as of research). Workaround shipped by open-vibe-island: at SessionStart the main app probes `focused terminal of selected tab of front window`, stores its AppleScript id, later focuses by id; fallbacks: working-directory match, title match (set CLAUDE_CODE_DISABLE_TERMINAL_TITLE to keep titles stable — trick from onikan27/claude-code-monitor). Feature-detect tty property at runtime for 1.4+.
- (b) WezTerm: PANE-LEVEL via CLI, no AppleScript. `wezterm cli list --format json` rows include pane_id, tab_id, window_id, tty_name, cwd, title, is_active -> map captured tty (or just use $WEZTERM_PANE captured at hook time) -> `wezterm cli activate-pane --pane-id N` (+ `wezterm cli activate-tab --tab-id N` if needed); then raise app with NSRunningApplication activate / `open -a WezTerm` (CLI activation does not always raise the OS window).
- (b) kitty: WINDOW(=pane)-LEVEL via remote control. Requires allow_remote_control yes (or password) AND a socket: listen_on unix:/tmp/kitty-{kitty_pid} in kitty.conf; then `kitten @ --to unix:/tmp/kitty-<KITTY_PID> focus-window --match id:<KITTY_WINDOW_ID>` (match fields: id, title, pid, cwd, cmdline, num, env, recent; `kitten @ ls` dumps the full window tree as JSON). focus-window raises the OS window and switches tab. Capture KITTY_WINDOW_ID + KITTY_LISTEN_ON/KITTY_PID at hook time. If remote control is disabled, only app-level activation is possible — onboarding must tell the user to add the config lines.
- (b) Alacritty: APP-LEVEL ONLY. No AppleScript dictionary, no tab model, `alacritty msg` only supports create-window/config. Focus = NSRunningApplication activate / open -b org.alacritty; per-window best-effort via Accessibility AXRaise matching window title (title can be set with OSC 2 shell integration).
- (b) Warp (dev.warp.Warp-Stable): APP-LEVEL ONLY today. No AppleScript (warpdotdev/Warp issues #1228, #3364). warp:// URI scheme only CREATES sessions: warp://action/new_window?path=..., warp://action/new_tab?path=..., warp://tab_config/<name> (docs.warp.dev/terminal/more-features/uri-scheme). Feature request #8611 (Feb 2026) for WARP_SESSION_ID env + warp://session/<id> deep link was closed referencing PR #11130 — runtime feature-detect: if getenv("WARP_SESSION_ID") exists at hook time, try `open warp://session/$WARP_SESSION_ID`, else NSRunningApplication activate. open-vibe-island additionally mines a per-pane UUID from Warp's local SQLite state but still cannot focus a specific pane with it.
- (b) tmux: `tmux list-panes -a -F '#{pane_id} #{pane_tty} #{session_name}:#{window_index}.#{pane_index}'` -> match on the claude tty -> `tmux select-window -t <session>:<window>` + `tmux select-pane -t %<pane_id>`; if the session isn't the one attached in the visible client: `tmux switch-client -c <client_tty> -t <session>` (exact argv confirmed in open-vibe-island: ["select-pane","-t",target], ["switch-client","-c",clientTTY,"-t",sessionName], ["select-window","-t",sessionWindow]). Always chain with focusing the hosting terminal via client_tty. Zellij analog: `zellij --session <name> action go-to-tab <n>`; cmux analog: Unix-socket JSON-RPC {"method":"surface.focus","params":{"surface_id":...}}.
- (c) VS Code/Cursor deep links: vscode://<publisher>.<extension>/<path>?<query> (and cursor:// — Cursor registers its own urlProtocol) is delivered to window.registerUriHandler() of that exact extension (authority MUST equal the extension id); onUri activation event wakes the extension. CAVEAT: with multiple IDE windows, the topmost/last-focused window handles the URI — mitigate by running `code -r <workspacePath>` / `cursor -r <workspacePath>` FIRST to raise the correct window, then firing the URI; or have the helper extension in every window write {windowPid, workspaceFolders, localhost port} to ~/Library/Application Support/<app>/vscode-windows/ and talk to the right window directly over localhost, skipping URIs entirely.
- (c) terminal-focus helper extension mechanics (Vibe Island pattern): UriHandler receives ?pid=<claudePid>&tty=...&cwd=...; iterate vscode.window.terminals; for each, `const shellPid = await terminal.processId` (pid of the terminal's shell process under VS Code's ptyhost); match if shellPid is an ancestor of claudePid (the app can pass the whole ancestor pid chain in the URI so the extension only does equality checks); on match `terminal.show(false)` reveals the terminal view, switches to that terminal instance, and focuses it. Secondary matchers: terminal.name, terminal.creationOptions.cwd.
- (c) Sideloading without marketplace: an unpacked folder copied to ~/.vscode/extensions/<publisher>.<name>-<version>/ (must contain package.json) is auto-discovered at next launch — extensions.json in that dir is a cache, not an allowlist (verified layout locally in /Users/mehdiguiard/.cursor/extensions: folders like anthropic.claude-code-2.1.214-darwin-arm64 beside extensions.json). More robust: ship a .vsix in app resources and run `code --install-extension x.vsix` / `cursor --install-extension x.vsix`; CLI lives at <App>.app/Contents/Resources/app/bin/{code|cursor} even when not on PATH. Requires a window reload to activate (prompt the user or `code -r` triggers it on new windows).
- (c) Official Claude Code IDE extension (inspected locally: /Users/mehdiguiard/.cursor/extensions/anthropic.claude-code-2.1.214-darwin-arm64/package.json + extension.js): activationEvents = ['onStartupFinished','onWebviewPanel:claudeVSCodePanel'] — NO onUri, so NO deep link can reach it directly. It contributes commands claude-vscode.focus, claude-vscode.blur, claude-vscode.sidebar.open, claude-vscode.terminal.open, claude-vscode.editor.open, claude-vscode.window.open — a helper extension CAN call vscode.commands.executeCommand('claude-vscode.focus') to raise the Claude panel. Two spawn paths found in extension.js: createTerminal({name:..., iconPath:...}) (terminal mode -> real tty via ptyhost -> fully detectable/focusable like any terminal) and an SDK path setting CLAUDE_CODE_ENTRYPOINT="sdk-ts" with stdio pipes (webview GUI mode -> NO tty, e_tdev=NODEV). Hooks still fire in both modes (same CLI underneath; hook stdin JSON has session_id, transcript_path, cwd, hook_event_name, permission_mode per code.claude.com/docs/en/hooks). Detection for panel sessions: CLAUDE_CODE_ENTRYPOINT=sdk-ts in hook env + ancestor chain ending in Code Helper (Plugin) under com.microsoft.VSCode or com.todesktop.230313mzl4w4u92 (Cursor's bundle id). Focus granularity: window-level only (code/cursor -r <cwd>), optionally + claude-vscode.focus via helper extension.
- (d) PERMISSIONS: sending Apple Events with hardened runtime/sandbox requires entitlement com.apple.security.automation.apple-events=true AND Info.plist NSAppleEventsUsageDescription (absent -> immediate errAEEventNotPermitted -1743, no prompt ever). TCC prompts once per (sender app, target app) pair at first send; grants live in kTCCServiceAppleEvents; denial is persistent and surfaces ONLY as -1743 on every send (no re-prompt). User remediation: System Settings > Privacy & Security > Automation (deep-linkable: open "x-apple.systempreferences:com.apple.preference.security?Privacy_Automation"). Reset: `tccutil reset AppleEvents <your-app-bundle-id>` (clears all targets for your sender) or bare `tccutil reset AppleEvents`. Preflight: AEDeterminePermissionToAutomateTarget(&addressDesc, typeWildCard, typeWildCard, askUserIfNeeded) -> noErr(granted) / -600 procNotFound(TARGET MUST BE RUNNING) / -1743(denied) / -1744 errAEEventWouldRequireUserConsent(not yet asked, askUserIfNeeded=false).
- (d) TCC attribution pitfall: run ALL AppleScript from the main notch app (NSAppleScript/OSAScript in-process, or its own osascript child). If the HOOK helper sent Apple Events, TCC would attribute them to the hook's responsible process (the terminal app itself), producing confusing 'Terminal wants to control iTerm2' prompts or silent failures. Therefore: hook helper only captures data and IPCs it to the app; the app performs probes/focus.
- (d) Accessibility fallback: separate TCC service (AXIsProcessTrustedWithOptions([kAXTrustedCheckOptionPrompt: true]); reset via tccutil reset Accessibility <id>). Viable for window-level raise on ANY app: AXUIElementCreateApplication(pid) -> kAXWindowsAttribute -> match kAXTitleAttribute -> AXUIElementPerformAction(win, kAXRaiseAction) + set kAXMainAttribute; NSRunningApplication.activate() needs no permission at all. NOT viable for per-tab targeting in GPU-rendered UIs (Warp, Alacritty, kitty draw custom tab bars with little/no AX tree). System Events UI scripting needs BOTH Automation(System Events) and Accessibility — avoid.
- Claude Code hook input (code.claude.com/docs/en/hooks): stdin JSON {session_id, prompt_id (v2.1.196+), transcript_path, cwd, permission_mode, hook_event_name, effort{level}}; relevant events SessionStart (capture anchor here), UserPromptSubmit, Stop, Notification, SessionEnd, CwdChanged (re-capture cwd); env adds CLAUDE_PROJECT_DIR; hooks run in both CLI and IDE-extension sessions. Register in ~/.claude/settings.json.
- Working open-source references: Octane0411/open-vibe-island (Swift, same architecture — key files Sources/OpenIslandApp/TerminalJumpService.swift, TerminalJumpTargetResolver.swift, Sources/OpenIslandCore/ClaudeHooks.swift; match priority: terminalSessionID > terminalTTY > workingDirectory > paneTitle > tmuxTarget > warpPaneUUID; VS Code family map {com.microsoft.VSCode: code, com.todesktop.230313mzl4w4u92: cursor, com.exafunction.windsurf: windsurf} invoked as `<cli> -r <workspacePath>`; generic fallback `open -b <bundleId>`); onikan27/claude-code-monitor (iTerm2/Terminal.app/Ghostty focus via AppleScript, Ghostty by title); vibeislandapp/vibe-island (closed-source app, community repo confirms terminal-focus VS Code extension + 20-terminal support incl. split panes).

## Recommendations
- Data captured at hook time (SessionStart + refreshed on CwdChanged/Stop) by a tiny native helper with zero TCC footprint: {session_id, transcript_path, cwd, own e_tdev tty via proc_pidinfo, full ancestor pid chain via pbi_ppid walk (pids + proc_name + proc_pidpath), env subset: TERM_PROGRAM, ITERM_SESSION_ID, KITTY_WINDOW_ID, KITTY_LISTEN_ON, KITTY_PID, WEZTERM_PANE, TMUX, TMUX_PANE, GHOSTTY_RESOURCES_DIR, WARP_IS_LOCAL_SHELL_SESSION, WARP_SESSION_ID, ALACRITTY_WINDOW_ID, VSCODE_INJECTION, CURSOR_TRACE_ID, CLAUDE_CODE_ENTRYPOINT, __CFBundleIdentifier} shipped as JSON over a unix socket to the main app. Never send Apple Events from the helper.
- Adapter interface: protocol TerminalFocusAdapter { var bundleIdentifiers: [String] { get }; func capability(for anchor: SessionAnchor) -> FocusGranularity /* .pane, .tab, .window, .app */; func focus(_ anchor: SessionAnchor) async throws }. Implement: TerminalAppAdapter + ITerm2Adapter (AppleScript by tty), GhosttyAdapter (AppleScript by stored focused-terminal id, cwd/title fallback; auto-upgrade to tty property when Ghostty >= 1.4), WezTermAdapter (cli list json tty_name -> activate-pane, then NSRunningApplication raise), KittyAdapter (kitten @ focus-window --match id:, degrade to app activation if no socket), TmuxAdapter (decorator: resolves pane + client_tty, then delegates outer focus to the host terminal's adapter), VSCodeFamilyAdapter (cli -r cwd, then vscode://<pub>.terminal-focus URI; executeCommand claude-vscode.focus for sdk-ts sessions), FallbackAdapter (NSRunningApplication.activate / open -b, optional AX window raise). Every focus() falls back down pane->tab->window->app and reports achieved granularity to the UI.
- Ship the terminal-focus IDE extension as a .vsix in app resources; install lazily the first time a session is detected inside VS Code/Cursor by invoking <App>.app/Contents/Resources/app/bin/{code|cursor} --install-extension, falling back to unpacking into ~/.vscode/extensions/<publisher>.terminal-focus-<version>/ and ~/.cursor/extensions/... ; in the URI pass the full ancestor pid chain so the extension only compares await terminal.processId against it and calls terminal.show(). Mitigate multi-window URI routing by always running `<cli> -r <workspace>` immediately before opening the URI (or adopt a per-window localhost port registry written by the extension).
- Permission onboarding: request nothing at install. On first jump to a given terminal, preflight with AEDeterminePermissionToAutomateTarget(askUserIfNeeded: true) while the target is running; on -1743 show inline UI linking to x-apple.systempreferences:com.apple.preference.security?Privacy_Automation and document `tccutil reset AppleEvents <your-bundle-id>` as support escape hatch. Include com.apple.security.automation.apple-events entitlement + NSAppleEventsUsageDescription now. Treat Accessibility as an optional, separately-gated enhancement used only for AX window-raise of Warp/Alacritty; app-level NSRunningApplication.activate needs no permission and must always be the final fallback.
- Expose per-terminal setup nudges in the UI: kitty requires allow_remote_control yes + listen_on unix:/tmp/kitty in kitty.conf; Ghostty users should keep macos-applescript enabled and (until 1.4) either rely on the SessionStart focused-terminal probe or set CLAUDE_CODE_DISABLE_TERMINAL_TITLE=1 for stable title matching; Warp users get app-level only today — feature-detect WARP_SESSION_ID/warp://session/<id> (warpdotdev/warp#8611, PR #11130) each release; tmux support must recursively focus the outer terminal via #{client_tty} before selecting window/pane.
- For sessions launched from the official Claude Code IDE panel (CLAUDE_CODE_ENTRYPOINT=sdk-ts, no tty): mark them with an IDE badge, focus via `code -r <cwd>` / `cursor -r <cwd>` (choose CLI by ancestor bundle id: com.microsoft.VSCode vs com.todesktop.230313mzl4w4u92), and if the helper extension is installed, follow with a URI that runs vscode.commands.executeCommand('claude-vscode.focus') to open the Claude panel itself — that is the maximum achievable granularity for panel sessions.
