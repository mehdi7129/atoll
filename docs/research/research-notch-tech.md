# notch-tech

## Summary
All serious macOS Dynamic-Island apps use the same core recipe: a borderless, non-activating, transparent NSPanel (or NSWindow) pinned top-center over the physical notch, at a window level above the menu bar, with collectionBehavior [.canJoinAllSpaces, .stationary, .fullScreenAuxiliary, .ignoresCycle] so it persists across Spaces and appears over fullscreen apps, hosting SwiftUI via NSHostingView. Notch geometry comes from NSScreen.safeAreaInsets.top (height) and frame.width minus auxiliaryTopLeftArea/auxiliaryTopRightArea widths (all macOS 12+; the auxiliary areas are nil on non-notch screens, which is also the canonical hasNotch test). Expand/collapse is done entirely in SwiftUI by animating a state enum (closed/popping/open) with spring animations inside a fixed-size window — the window frame itself is never animated — using either SwiftUI .onHover with delayed-open/delayed-close Tasks (boring.notch, DynamicNotchKit) or global+local NSEvent monitors with rect hit-testing against the computed notch rect (NotchDrop). Non-notch Macs and external displays are handled by drawing a fake notch pill (e.g. 150x28 or 185 x menu-bar-height) or a floating rounded HUD (DynamicNotchKit's .floating style); multi-screen support means one window + view-model per NSScreen keyed by a stable displayUUID from CGDisplayCreateUUIDFromDisplayID, rebuilt on NSApplication.didChangeScreenParametersNotification. boring.notch (10k stars, GPL-3.0, the most feature-complete reference) additionally uses private CGS/SkyLight APIs (CGSSpaceCreate/CGSSpaceSetAbsoluteLevel/CGSAddWindowsToSpaces, and Lakr233/SkyLightWindow's SLS delegation) for lock-screen display — the same trick commercial Alcove markets — plus TheBoredTeam/MacroVisionKit for per-screen fullscreen detection to optionally hide the notch UI. DynamicNotchKit v1.1.0 (MIT, macOS 13+, Swift 6) is the cleanest reusable Swift package with an async expand()/compact()/hide() API, but is geared to transient HUD/notification use rather than a persistent interactive notch app. The most robust 2025-2026 approach is to own the window layer (boring.notch-style NSPanel config), compute geometry via an NSScreen extension with menu-bar-height fallback, drive all animation from SwiftUI state, keep canBecomeKey false by default, and avoid private APIs unless lock-screen support is a hard requirement.

## Key findings
- Open-source references: TheBoredTeam/boring.notch (GPL-3.0, ~10,075 stars, v2.7.3, very active — full notch app: media, shelf, HUD replacement), MrKai77/DynamicNotchKit (MIT, 438 stars, v1.1.0 2026-04, SPM package, macOS 13+, swift-tools 6.0), Lakr233/NotchDrop (MIT, 2,074 stars — file-drop shelf), plus helper packages Lakr233/SkyLightWindow (MIT, lock-screen/above-everything windows) and TheBoredTeam/MacroVisionKit (fullscreen detection). Commercial: NotchNook (lo.cafe, $25/lifetime, hover-click-swipe open, simulated notch on non-notch displays, now-playing stays above fullscreen apps) and Alcove (tryalcove.com, Henrik Ruscon, $17, macOS 15+, brew cask 'alcove', unique lock-screen visibility + system HUD replacement).
- Canonical panel config (boring.notch BoringNotchWindow.swift): NSPanel subclass, styleMask [.borderless, .nonactivatingPanel, .utilityWindow, .hudWindow], isFloatingPanel=true, isOpaque=false, backgroundColor=.clear, hasShadow=false, isMovable=false, isReleasedWhenClosed=false, level = .mainMenu + 3, collectionBehavior = [.fullScreenAuxiliary, .stationary, .canJoinAllSpaces, .ignoresCycle], canBecomeKey=false, canBecomeMain=false, show with orderFrontRegardless(). Variants: NotchDrop uses NSWindow with level .statusBar + 8 and canBecomeKey=true (needed for drag-drop/text interaction, calls NSApp.activate(ignoringOtherApps:) on open); DynamicNotchKit uses NSPanel level .screenSaver, styleMask [.borderless, .nonactivatingPanel], collectionBehavior [.canJoinAllSpaces, .stationary], canBecomeKey=true.
- Notch geometry (all projects identical, macOS 12+ APIs): hasNotch = screen.auxiliaryTopLeftArea != nil && screen.auxiliaryTopRightArea != nil (both are NSRect?, nil without camera housing); notchHeight = screen.safeAreaInsets.top; notchWidth = screen.frame.width - auxiliaryTopLeftArea!.width - auxiliaryTopRightArea!.width (boring.notch adds +4pt slop); notchFrame = centered: x = frame.midX - w/2, y = frame.maxY - h. Menu-bar-height fallback for non-notch screens: frame.maxY - visibleFrame.maxY. Built-in display test: CGDisplayIsBuiltin(CGDirectDisplayID(deviceDescription[NSDeviceDescriptionKey("NSScreenNumber")])) == 1 (NotchDrop Ext+NSScreen.swift).
- Non-notch Macs / external displays: NotchDrop draws a fake notch 150x28; boring.notch uses configurable nonNotchHeight (or match-menu-bar mode) with default width 185; DynamicNotchKit's DynamicNotchStyle enum { .notch(topCornerRadius:15, bottomCornerRadius:20), .floating(cornerRadius:20), .auto } — .auto picks per-screen via screen.hasNotch, and .floating renders a slide-down rounded HUD (NotchlessView with VisualEffectView material .popover, blendingMode .behindWindow) instead of a notch extension.
- Window-sizing strategy: never animate the NSWindow frame. boring.notch allocates a fixed window of openNotchSize 640x190 + 20pt shadowPadding, top-centered (setFrameOrigin x: screenFrame.origin.x + width/2 - windowWidth/2, y: origin.y + height - windowHeight); NotchDrop allocates a full-screen-width, 200pt-tall strip pinned to the top; DynamicNotchKit allocates screen.width/2 x screen.height/2. All expansion/collapse is SwiftUI content animation inside the oversized transparent window; clicks pass through transparent SwiftUI regions (hit area defined with .contentShape(Rectangle()) on the notch layout), so window.ignoresMouseEvents is not used by any of the three (it can still be toggled true when collapsed for guaranteed click-through).
- Hover/expand mechanics, approach A (boring.notch ContentView.swift handleHover): SwiftUI .onHover on the notch shape; on hover-in set isHovering with spring animation, trigger haptics (.sensoryFeedback(.alignment) / NSHapticFeedbackManager), then Task.sleep(Defaults[.minimumHoverDuration]) before vm.open() (guarding state unchanged); on hover-out Task.sleep(100ms) grace period then vm.close(). Open animation .spring(response: 0.42, dampingFraction: 0.8), close .spring(response: 0.45, dampingFraction: 1.0); also pan-gesture (swipe down to open, up to close) via custom PanGesture extension.
- Hover/expand mechanics, approach B (NotchDrop EventMonitor/NotchViewModel+Events): paired NSEvent.addGlobalMonitorForEvents + addLocalMonitorForEvents for .mouseMoved/.leftMouseDown/.leftMouseDragged/.flagsChanged feeding Combine subjects; state machine closed/popping/opened; open on mouseDown inside deviceNotchRect.insetBy(dx: -4, dy: -4); 'popping' preview when mouse hovers the notch rect; close on click outside notchOpenedRect (600x160); animation .interactiveSpring(duration: 0.5, extraBounce: 0.25, blendDuration: 0.125); haptics throttled 0.5s .levelChange. Global mouseMoved monitors need no privacy permission; the local monitor covers events over your own window.
- DynamicNotchKit public API: DynamicNotch<Expanded, CompactLeading, CompactTrailing>(hoverBehavior:style:expanded:compactLeading:compactTrailing:) with async expand(on screen:), compact(on:), hide(); iOS-Dynamic-Island-style compact leading/trailing views rendered either side of a Spacer exactly notchSize.width wide; DynamicNotchHoverBehavior OptionSet [.keepVisible, .hapticFeedback, .increaseShadow]; DynamicNotchInfo/DynamicNotchProgress convenience classes; anti-glitch pattern: start withAnimation state change BEFORE orderFrontRegardless(), then fade window alphaValue 0->1 (NSAnimationContext 0.15s easeOut) and fade out before close; animations .bouncy(duration:0.4) open / .smooth(0.4) close / .snappy(0.4) compact-expand conversion.
- Notch shape drawing: custom SwiftUI Shape (NotchShape in both DynamicNotchKit and boring.notch) — top corners curve outward (quad curve from screen edge into the notch), bottom corners rounded inward, with animatableData = AnimatablePair(topCornerRadius, bottomCornerRadius) so corner radii animate during expand; radii: closed (top 6, bottom 14), expanded boring.notch (19, 24) / DynamicNotchKit (15, 20); rule of thumb outerRadius = innerRadius + padding; content is black-filled Rectangle masked by the shape so it visually merges with the physical notch (boring.notch forces appearance = NSAppearance(named: .darkAqua)).
- Showing above fullscreen apps: .fullScreenAuxiliary in collectionBehavior lets the panel join native fullscreen Spaces; combined with level > .mainMenu (e.g. .mainMenu+3, .statusBar+8, or .screenSaver) the notch UI renders over fullscreen apps. Remaining issues (boring.notch issues #396/#803): the auto-revealed menu bar overlaps on hover, and users often want the notch HIDDEN in fullscreen instead — solved via TheBoredTeam/MacroVisionKit (SPM, from: "0.2.0"): FullScreenMonitor.shared.spaceChanges() async stream of SpaceInfo {screenUUID, runningApps} to hide per-screen when a fullscreen app (optionally only the now-playing app) is active.
- Lock-screen / above-absolutely-everything (private APIs, not App-Store-safe): boring.notch CGSSpace.swift (from avaidyam/Parrot) declares via @_silgen_name: _CGSDefaultConnection, CGSSpaceCreate(cid, 0x1, nil), CGSSpaceSetAbsoluteLevel(cid, space, 2147483647), CGSAddWindowsToSpaces/CGSRemoveWindowsFromSpaces, CGSShowSpaces/CGSHideSpaces — windows added to this custom space float over everything; Lakr233/SkyLightWindow (macOS 11+) wraps the same via SkyLightOperator.shared.delegateWindow(_:) and .moveToSky() SwiftUI modifier, and boring.notch dlopen()s /System/Library/PrivateFrameworks/SkyLight.framework to call SLSRemoveWindowsFromSpaces for undelegation; boring.notch enables SkyLight only while DistributedNotificationCenter 'com.apple.screenIsLocked'/'com.apple.screenIsUnlocked' says the screen is locked. This is how Alcove's lock-screen feature works in practice.
- Multi-screen: boring.notch keeps windows: [String: NSWindow] and viewModels: [String: BoringViewModel] keyed by displayUUID (CGDisplayCreateUUIDFromDisplayID -> CFUUIDCreateString, cached and rebuilt on NSApplication.didChangeScreenParametersNotification); on screen-config change it compares UUID sets + frames, closes windows for vanished screens, creates for new ones, repositions the rest (temporarily alphaValue=0 to hide repositioning); optional single-display mode follows NSScreen.main with automaticallySwitchDisplay; NSWindow.didChangeScreenNotification observed per-window; DynamicNotchKit re-initializes its window on the same notification; NSScreen.screenWithMouse helper: screens.first { NSMouseInRect(NSEvent.mouseLocation, $0.frame, false) }.
- App-shell details: SwiftUI @main App with MenuBarExtra scene + @NSApplicationDelegateAdaptor (no regular windows; accessory/LSUIElement activation policy); applicationShouldTerminateAfterLastWindowClosed -> false; hide from screen recording via window.sharingType = .none (boring.notch toggles with a default); boring.notch dependencies worth reusing: Defaults, KeyboardShortcuts, Sparkle, LaunchAtLogin (NotchDrop).
- Gotchas: use orderFrontRegardless() not makeKeyAndOrderFront for nonactivating panels (NotchDrop uses makeKeyAndOrderFront because it wants key status); .nonactivatingPanel means clicking the notch never steals focus from the frontmost app — required for a good UX; .stationary + .ignoresCycle keep the panel out of Exposé and Cmd-Tab window cycling; isReleasedWhenClosed=false prevents crashes on close/reopen; NSHostingView content should over-paint (padding -50 black background in DynamicNotchKit) because spring animations overshoot the mask.

## Recommendations
- Recommended architecture (2025-2026): write your own window layer (~100 lines) modeled on boring.notch rather than depending on a full app: NSPanel subclass with styleMask [.borderless, .nonactivatingPanel], level .statusBar + 3 (or .screenSaver if you must beat other utilities), collectionBehavior [.canJoinAllSpaces, .stationary, .fullScreenAuxiliary, .ignoresCycle], clear/non-opaque/no-shadow, canBecomeKey=false, one panel per screen keyed by displayUUID, fixed frame equal to the maximum expanded size top-centered, all animation in SwiftUI state.
- Use MrKai77/DynamicNotchKit (SPM: https://github.com/MrKai77/DynamicNotchKit, macOS 13+) directly if the need is transient notch HUDs/notifications — its async expand/compact/hide API, .auto notch/floating style fallback, and anti-flicker window choreography are production-ready; fork or re-implement for a persistent, always-interactive notch app since it tears down its window when hidden.
- Geometry: implement the standard NSScreen extension (hasNotch via auxiliaryTopLeftArea/auxiliaryTopRightArea nil-check; size from safeAreaInsets.top and frame minus auxiliary widths; fallback pill 185-300pt wide x menu-bar height frame.maxY - visibleFrame.maxY for non-notch/external displays) and recompute + rebuild windows on NSApplication.didChangeScreenParametersNotification, diffing displayUUID sets like boring.notch does.
- Interaction: prefer SwiftUI .onHover + .contentShape on the notch shape with a configurable minimum-hover-duration Task before expanding and a ~100ms grace Task before collapsing (boring.notch pattern); add a NotchDrop-style global .leftMouseDown monitor only for 'click outside to close'; drive expand/collapse with a spring (open .spring(response:0.42,dampingFraction:0.8) or .bouncy(0.4); close .smooth/.spring(response:0.45,dampingFraction:1.0)) and animate NotchShape corner radii via animatableData.
- Focus handling: keep the panel non-activating (canBecomeKey=false, orderFrontRegardless) so the user's app keeps focus; if you later need text input inside the expanded notch, override canBecomeKey=true and call makeKey only while expanded (DynamicNotchKit/NotchDrop precedent).
- Fullscreen: .fullScreenAuxiliary + high level already shows the overlay above fullscreen apps with zero extra work; add TheBoredTeam/MacroVisionKit only if you want the opposite behavior (auto-hide in fullscreen, per screen). Avoid private CGS/SkyLight APIs (CGSSpaceCreate, SkyLightWindow) unless lock-screen display is a hard requirement — they are unsandboxable, App-Store-ineligible, and break across macOS majors; if used, gate them behind the com.apple.screenIsLocked distributed notifications like boring.notch.
- License caution: boring.notch is GPL-3.0 — copy its architecture/ideas, not its code, for a proprietary app; DynamicNotchKit, NotchDrop, and SkyLightWindow are MIT. Study NotchNook and Alcove for UX benchmarks (hover/click/swipe open modes, HUD replacement, simulated notch on external displays, fade-out when idle).
