# usage-quota

## Summary
Verified on this machine (Claude Code 2.1.214, macOS): the statusLine command receives, on stdin, a JSON object that includes an authoritative `rate_limits` block — `{"five_hour":{"used_percentage":2,"resets_at":1784401800},"seven_day":{"used_percentage":12,"resets_at":1784426400}}` (percent 0-100, Unix epoch seconds) — sourced server-side from `anthropic-ratelimit-unified-5h/7d-utilization/-reset` response headers the CLI caches on every API response, so it is exact, not estimated. It fires after each assistant message, /compact, permission-mode or vim-mode change, debounced 300 ms, plus an optional `statusLine.refreshInterval` (seconds, min 1) timer; `rate_limits` appears only for Pro/Max subscribers after the first API response and is absent in ~23% of invocations (Vibe Island telemetry), and only ONE statusLine command is allowed, so coexistence requires a tee-wrapper that caches the payload then pipes stdin to the user's original command (Vibe Island installs a "usage bridge" into the slot and refuses to install over an unrecognized custom statusline; Open Island writes ~/.open-island/bin/open-island-statusline caching to /tmp/open-island-rl.json; ccusage simply occupies the slot). `rate_limit_event` (with rate_limit_info: status/rateLimitType/resetsAt/utilization) exists only in stream-json output for -p/SDK sessions since CLI 2.1.45, fires only on status CHANGES, omits utilization when status is "allowed" (issue #50518), and never fires for interactive TUI sessions; no hook event carries rate-limit data. The fallback used by Vibe Island/ClaudeBar/xbar plugins is `GET https://api.anthropic.com/api/oauth/usage` with `Authorization: Bearer <accessToken>` + `anthropic-beta: oauth-2025-04-20` + `anthropic-version: 2023-06-01`, the token read from Keychain item service "Claude Code-credentials" (account = $USER, JSON `{"claudeAiOauth":{accessToken,refreshToken,expiresAt(ms),scopes,subscriptionType}}`; ~/.claude/.credentials.json only on Linux/fallback — absent on this Mac); response is `{five_hour:{utilization:<0-100>,resets_at:<ISO-8601>},seven_day:{...},seven_day_opus,seven_day_sonnet,seven_day_oauth_apps,extra_usage:{is_enabled,monthly_limit,used_credits,utilization,currency},limits:[{kind,group,percent,resets_at,scope}]}` — note unit mismatches vs statusline (percent vs fraction, ISO vs epoch). ToS reality: since Jan 9 2026 Anthropic server-blocks consumer OAuth tokens in third-party INFERENCE tools and the Feb 19 2026 docs state any third-party use of these tokens violates the Consumer ToS; read-only usage polling remains unblocked and widely shipped (ClaudeBar, xbar, Vibe Island) but is undocumented, technically non-compliant, and revocable anytime — and a third-party app must NEVER use the refreshToken (rotation desyncs Claude Code, forcing re-login; CodexBar #1161). Local JSONL cost estimation (ccusage) demonstrably drifts (issues #247/#326/#483/#487/#779/#916): limits are account-wide across devices and surfaces, the server's weighting/points model is unpublished, and 5-hour "blocks" are heuristics — unusable for trustworthy percentages. Recommended architecture: statusline tee-wrapper cache as primary (ToS-clean, exact), oauth/usage polling (60s-5min, re-read Keychain on 401, never refresh) as opt-in secondary for idle/cold-start coverage, stale-with-age-indicator then hide as degradation; skip JSONL estimation for limit display entirely.

## Key findings
- (a) EXACT statusline stdin JSON, verified live on CLI 2.1.214 (captured from ~/.claude/scripts/statusline/data/last_payload.txt on this machine): top-level keys session_id, transcript_path, cwd, prompt_id, effort:{level}, session_name, model:{id,display_name}, workspace:{current_dir,project_dir,added_dirs}, version, output_style:{name}, cost:{total_cost_usd,total_duration_ms,total_api_duration_ms,total_lines_added,total_lines_removed}, context_window:{total_input_tokens,total_output_tokens,context_window_size,current_usage:{input_tokens,output_tokens,cache_creation_input_tokens,cache_read_input_tokens},used_percentage,remaining_percentage}, exceeds_200k_tokens, fast_mode, thinking:{enabled}, rate_limits:{five_hour:{used_percentage:<0-100>,resets_at:<unix epoch seconds>},seven_day:{same}}
- (a) rate_limits provenance (from binary disassembly of ~/.local/share/claude/versions/2.1.214): the CLI parses response headers anthropic-ratelimit-unified-{5h,7d,7d_oi,overage}-utilization (fraction 0-1) and -reset (epoch seconds) on every API response into a cached map; statusline payload computes used_percentage = utilization*100. Also parsed: anthropic-ratelimit-unified-status (allowed|allowed_warning|rejected), -reset, -representative-claim, -fallback, -overage-status/-reset/-in-use/-disabled-reason. So statusline percentages are server-authoritative and account-wide, not local estimates.
- (a) Firing cadence (official docs code.claude.com/docs/en/statusline): runs after each new assistant message, after /compact, on permission-mode change, on vim-mode toggle; debounced 300 ms; in-flight execution cancelled if a new update triggers. Optional settings field statusLine.refreshInterval (seconds, min 1, verified in 2.1.214 binary schema) re-runs it on a fixed timer during idle. Docs caveats: rate_limits appears ONLY for Claude.ai Pro/Max subscribers and only after the first API response of a session; five_hour/seven_day may be independently absent (use jq '// empty'). Vibe Island issue #9 measured rate_limits missing in ~23% of statusline calls and used an mtime<60s last-known-good cache.
- (a) Single-slot strategies observed in the wild: (1) ccusage occupies the slot itself ({"statusLine":{"type":"command","command":"bun x ccusage statusline","padding":0}}); (2) Vibe Island installs a managed 'usage bridge' command into statusLine and, after early versions clobbered users' ccstatusline configs (issue #107), now refuses to auto-install when an existing custom statusLine is present (issues #137, #153 — its existence check failed on $HOME-style paths because it uses stringByExpandingTildeInPath, fixed ~v1.0.40); (3) Open Island (Octane0411/open-vibe-island) writes a managed statusLine.command at ~/.open-island/bin/open-island-statusline that caches rate_limits into /tmp/open-island-rl.json and 'refuses to overwrite an existing custom status line automatically'. The correct chaining pattern: wrapper script tees stdin JSON to a cache file (or Unix socket), then pipes the identical JSON to the user's saved original command and echoes its stdout — pure passthrough, invisible to the user.
- (b) rate_limit_event: emitted in stream-json output only (-p / Agent SDK sessions), added ~CLI 2.1.45; shape {type:'rate_limit_event', rate_limit_info:{status:'allowed'|'allowed_warning'|'rejected', resetsAt?:number, rateLimitType?:'five_hour'|'seven_day'|'seven_day_opus'|'seven_day_sonnet'|'seven_day_overage_included'|'overage', utilization?:number, overageStatus?, overageResetsAt?, overageDisabledReason?, isUsingOverage?, surpassedThreshold?}, uuid, session_id} (schema extracted verbatim from 2.1.214 binary). Critical limits: fires only on status CHANGES, and utilization is omitted when status='allowed' (anthropics/claude-code#50518) — cannot drive a live percentage gauge. Interactive TUI sessions produce no stream-json, so this channel does not exist for them; conversely the statusline never renders in headless/stream-json mode (Open Island docs note Desktop/headless sessions never update statusline-fed caches). No hook event (PreToolUse/PostToolUse/SessionStart/Stop/Notification/etc.) carries rate-limit fields — statusline is the only push channel in interactive mode. Also relevant: {type:'system',subtype:'api_retry',error_status:429,error:'rate_limit'} events exist in stream-json for transient 429s.
- (c) Usage endpoint (verified in CLI binary — the CLI's own /usage panel calls it): GET https://api.anthropic.com/api/oauth/usage with headers 'Authorization: Bearer <accessToken>', 'anthropic-beta: oauth-2025-04-20', 'anthropic-version: 2023-06-01', 'Content-Type: application/json' (CLI uses 5s timeout, retries once after 401→refresh). Response (zod schema from binary + working xbar plugin): {five_hour:{utilization:<number 0-100>, resets_at:<ISO-8601 string|null>}, seven_day:{...}, seven_day_opus?, seven_day_sonnet?, seven_day_oauth_apps?, extra_usage:{is_enabled,monthly_limit,used_credits,utilization,currency,disabled_reason}?, limits:[{kind,group,percent,resets_at,scope:{model:...}}]?}. UNIT TRAP: oauth endpoint returns utilization as 0-100 percent + ISO-8601 resets_at; statusline gives used_percentage 0-100 + epoch seconds; raw headers give fraction 0-1 + epoch seconds. Vibe Island changelog explicitly 'unified reset-time parsing across statusline cache and OAuth schemas'.
- (c) Credential storage on macOS (verified locally): login-keychain generic-password item, service AND label 'Claude Code-credentials', account = macOS username; secret is JSON {"claudeAiOauth":{"accessToken":"sk-ant-oat01-...","refreshToken":"sk-ant-ort01-...","expiresAt":<ms epoch>,"scopes":[...],"subscriptionType":"max"|"pro"|...}}. ~/.claude/.credentials.json is the Linux/fallback path only (absent on this Mac). Shell read: security find-generic-password -s "Claude Code-credentials" -w. A sandboxed/native app using SecItemCopyMatching triggers the macOS keychain consent prompt; worse, Claude Code rewrites the item on every token refresh, resetting the ACL so 'Always Allow' grants are repeatedly lost (anthropics/claude-code#41026) — most tools shell out to /usr/bin/security instead. Token refresh endpoint: POST https://platform.claude.com/v1/oauth/token, client_id 9d1c250a-e61b-44d9-88ed-5944d1962f5e (both in binary). DO NOT refresh from a third-party app: Anthropic rotates the refresh token on use, so an external refresh desyncs Claude Code's stored refreshToken → invalid_grant → forced /login (CodexBar#1161, #1844). Correct pattern (ClaudeBar): read-only, re-read Keychain on 401 and let Claude Code do the refreshing; ClaudeBar polls ~1/min, xbar plugin defaults 5 min and warns shorter intervals can rate-limit.
- (c) ToS/risk (honest): Consumer ToS §3.7 has long banned this; enforcement began Jan 9 2026 with server-side blocks of consumer OAuth tokens in third-party inference tools (OpenCode, OpenClaw etc.), and a Feb 19 2026 docs clarification states: 'Using OAuth tokens obtained through Claude Free, Pro, or Max accounts in any other product, tool, or service — including the Agent SDK — is not permitted and constitutes a violation of the Consumer Terms of Service.' The text does not carve out read-only usage polling; to date enforcement targets inference (blocked requests, some account bans for 'gaming pricing structure'), and usage-monitor apps (ClaudeBar, Vibe Island, xbar plugins, many menubar clones) continue operating unblocked as of Jul 2026 — but the endpoint is undocumented, can move/401 anytime, and a strict reading makes it non-compliant. Ship it as an optional, clearly-labeled 'advanced' data source that is off by default or degraded-gracefully, never as the app's only mechanism.
- (d) ccusage/JSONL drift causes (issues ryoppippi/ccusage #247 #326 #483 #487 #779 #916; anthropics/claude-code#54750): (1) limits are ACCOUNT-wide — Claude web/desktop/mobile chat, Claude Code on other machines, and cloud sessions all consume the same windows but never appear in local ~/.claude/projects/*.jsonl; (2) Anthropic's real limiter is an unpublished weighted/points system (model multipliers, cache weighting), while ccusage computes USD via LiteLLM prices — different metric entirely; (3) ccusage's 5-hour 'billing blocks' are heuristics (first-message-rounded-to-hour), not the server's actual window anchor, so resets misalign; (4) dedup/missing-entry bugs (resumed sessions, sidechains/subagents, server-side tool tokens absent from JSONL); reported outcomes: hitting real limits while ccusage shows headroom (#483), Claude /usage 22% vs ccusage 6.9% (#916), ~20% underestimates vs actual charges (#779), session exhausted at 35.6M local tokens with no local explanation (#54750). Conclusion: JSONL math is fine for cost curiosity/burn-rate, unusable for trustworthy limit percentages.
- (e) Verified-good architecture for a native macOS app: PRIMARY = statusline tee-wrapper (exact, ToS-clean, zero credentials): on setup read ~/.claude/settings.json; if statusLine exists, persist the original command and install a wrapper that (i) appends the stdin JSON to an app cache (e.g. ~/Library/Application Support/<app>/rl-cache.json or a Unix socket), (ii) execs the original command with the same stdin, passing through stdout — user's statusline unchanged; if none exists, install wrapper alone (may print nothing or a minimal line). Set refreshInterval:60 so caches stay warm while sessions idle. SECONDARY (opt-in) = /api/oauth/usage poll every 60-300s via /usr/bin/security read of 'Claude Code-credentials', read-only, re-read on 401, never touch refreshToken — covers cold start, non-CLI usage (claude.ai chat) and idle gaps. DEGRADATION: fresh statusline cache (<60s) → oauth poll → stale cache with 'updated Xs ago' badge (Vibe Island v1.0.21: 'usage data never discarded — stale data stays visible with aging indicator') → hide expired countdowns (resets_at < now) → if neither source, show 'open a Claude Code session to populate' instead of JSONL guesses. Countdown = resets_at − now rendered locally; parse both epoch-seconds and ISO-8601.

## Recommendations
- Build the statusline tee-wrapper as the core data path: it is the only exact, account-wide, ToS-clean source available to a local app during interactive sessions; cache every payload's rate_limits with a received-at timestamp and treat <60s as fresh (Vibe Island's own threshold).
- Handle the single slot defensively: never silently overwrite statusLine (Vibe Island issue #107 backlash); detect existing commands including '$HOME'-style paths by expanding env vars through a shell, not NSString tilde expansion (issue #153); chain by exec'ing the saved original command with stdin passthrough; offer one-click uninstall restoring the original settings.json verbatim.
- Set statusLine.refreshInterval to ~60 in the wrapper install so rate_limits keep flowing while the session is idle; still expect gaps — the field is absent early-session and in ~23% of calls, so always render from cache, not from the latest payload.
- Add /api/oauth/usage polling only as a clearly-labeled optional enhancer: shell out to /usr/bin/security find-generic-password -s 'Claude Code-credentials' -w (avoids SecItem ACL prompts that reappear every token refresh per claude-code#41026), send Authorization Bearer + anthropic-beta: oauth-2025-04-20 + anthropic-version: 2023-06-01, poll 60-300s, on 401 re-read the Keychain once and back off; NEVER call the refresh endpoint (rotation bricks the user's Claude Code login, CodexBar#1161); disclose the ToS gray-zone in the UI/README like ClaudeBar does.
- Do not use rate_limit_event or hooks for percentages: no hook carries rate-limit data, and rate_limit_event exists only in stream-json (-p/SDK) sessions, fires only on status changes, and omits utilization while status='allowed' (claude-code#50518) — at most subscribe to it in SDK-driven flows to catch 'allowed_warning'/'rejected' transitions instantly.
- Do not derive limit percentages from local JSONL (ccusage approach): account-wide consumption, unpublished weighting, and heuristic 5h block anchoring make >100%/false-headroom drift structural, not fixable; if you show JSONL numbers at all, label them 'estimated local cost', never as the limit gauge.
- Normalize units at one boundary: statusline used_percentage(0-100)+epoch-seconds, oauth utilization(0-100)+ISO-8601, headers fraction(0-1)+epoch-seconds; store internally as fraction + Date, auto-hide countdowns whose resets_at has passed, and show a data-age indicator when serving stale values.
