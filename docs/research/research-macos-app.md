# macos-app

## Summary
For a solo dev shipping a small macOS SwiftUI utility outside the App Store in 2025-2026, the pragmatic stack is: a plain Xcode project (macOS App template, SwiftUI lifecycle) with minimum deployment target macOS 14.0, a MenuBarExtra scene with .menuBarExtraStyle(.window) plus LSUIElement=YES for a menu-bar-only app, SMAppService.mainApp for launch-at-login, App Sandbox OFF (legal for direct distribution and required in practice to spawn the user's claude CLI) with Hardened Runtime ON, a localhost HTTP server via FlyingFox or Network.framework's NWListener, subprocess spawning via Process running /bin/zsh -l -c (login shell) with a cached `command -v claude` path resolution plus manual PATH augmentation covering ~/.local/bin, /opt/homebrew/bin and /usr/local/bin, and Sparkle 2.8 via SPM for auto-updates with an appcast hosted on GitHub Pages/Releases. The paid Apple Developer Program ($99/yr) is effectively mandatory: macOS 15 Sequoia removed the Control-click→Open Gatekeeper bypass, so an unsigned/un-notarized app forces users through System Settings > Privacy & Security > "Open Anyway" with an "Apple could not verify this is free of malware" scare dialog — fatal for casual-user conversion. With the paid account the flow is: Developer ID Application certificate + Hardened Runtime, `xcrun notarytool store-credentials` once, `xcrun notarytool submit App.zip --keychain-profile notary --wait` (1-3 min), then `xcrun stapler staple`, distributing a notarized .dmg or ditto-zipped .app. macOS 14 as min target covers everything needed (MenuBarExtra and SMAppService are 13+, @Observable and SettingsLink are 14+); macOS 15 is only required for windowStyle(.plain), WindowDragGesture, pushWindow, and defaultWindowPlacement — none needed for a MenuBarExtra utility. A pure SwiftPM executable can run SwiftUI but produces a bare binary with no Info.plist, entitlements, icon, or signing/notarization path, so it is unsuitable for this app; use Xcode project for the app shell and optionally a local SPM package for logic.

## Key findings
- Gatekeeper without paid account: macOS 15 Sequoia (and later, incl. macOS 26 Tahoe) removed the Control-click/right-click 'Open' override for unsigned or un-notarized apps. The only user path is: attempt launch, dismiss the 'Apple could not verify ... is free of malware' dialog, then System Settings > Privacy & Security > 'Open Anyway' (one time per app; requires local admin approval). Power users can instead run `xattr -d com.apple.quarantine /Applications/App.app`. On Apple Silicon, all binaries must be at least ad-hoc signed (Xcode 'Sign to Run Locally' does this automatically).
- Paid path (recommended): Apple Developer Program $99/yr → 'Developer ID Application' certificate. Notarization requires Hardened Runtime enabled (ENABLE_HARDENED_RUNTIME=YES); App Sandbox is NOT required for notarization or direct distribution. CLI workflow: (1) one-time `xcrun notarytool store-credentials "notary" --apple-id <id> --team-id <TEAMID> --password <app-specific-password>` (or App Store Connect API key); (2) `ditto -c -k --keepParent MyApp.app MyApp.zip`; (3) `xcrun notarytool submit MyApp.zip --keychain-profile notary --wait` (~1-3 min); (4) `xcrun stapler staple MyApp.app` (staple the .app and/or the .dmg — the zip you submitted cannot be stapled, re-zip the stapled app or notarize+staple the .dmg directly). altool is dead; notarytool (Xcode 13+) is the only supported tool.
- Menu-bar-only app: SwiftUI `MenuBarExtra` scene (macOS 13+) with `.menuBarExtraStyle(.window)` for a rich popover UI (`.menu` for a plain dropdown). Set Info.plist `LSUIElement` = YES ('Application is agent (UIElement)') to hide the Dock icon and app switcher entry. Caveats: an app whose only scene is a MenuBarExtra terminates if the user Cmd-drags the icon off the menu bar; you must provide an explicit Quit control; MenuBarExtra has no right-click/option-click distinction — drop to NSStatusItem via NSApplicationDelegateAdaptor if you need that. Add a `Settings` scene and open it with `SettingsLink` (macOS 14+).
- Launch at login: `import ServiceManagement`; `SMAppService.mainApp.register()` / `.unregister()` (macOS 13+; replaces deprecated SMLoginItemSetEnabled and the LaunchAtLogin helper-bundle hack). Always read live state via `SMAppService.mainApp.status == .enabled` instead of caching in UserDefaults, because users can toggle it in System Settings > General > Login Items; macOS shows a 'login item added' notification on register. Default it to OFF and gate behind an explicit user toggle.
- Local HTTP server: for direct distribution, simply turn App Sandbox OFF — no entitlement needed to bind localhost, and notarization still passes (Hardened Runtime only). If you keep the sandbox (mandatory for App Store), you need `com.apple.security.network.server` = true to listen and `com.apple.security.network.client` for outgoing connections. Library options: Network.framework NWListener (zero-dependency), FlyingFox https://github.com/swhitty/FlyingFox (async/await, SPM, ideal for small embedded servers), Hummingbird https://github.com/hummingbird-project/hummingbird (SwiftNIO, heavier), Swifter https://github.com/httpswift/swifter (aging), Vapor (overkill).
- Spawning the claude CLI: GUI apps inherit launchd's minimal PATH (/usr/bin:/bin:/usr/sbin:/sbin), so `claude` won't be found. Pattern confirmed on Swift Forums: `let p = Process(); p.executableURL = URL(fileURLWithPath: "/bin/zsh"); p.arguments = ["-l", "-c", command]` — the login shell sources /etc/zprofile (path_helper) and ~/.zprofile. Gotcha: PATH exported only in ~/.zshrc is NOT picked up by `-l -c` (zshrc is interactive-only); adding `-i` risks hangs. Robust approach: resolve once with `/bin/zsh -l -c 'command -v claude'`, cache the absolute path, and also probe known install locations: ~/.local/bin/claude (native installer symlink into ~/.local/share/claude/versions/ — the current recommended install), /opt/homebrew/bin/claude, /usr/local/bin/claude, npm-global (/usr/local/lib/node_modules/@anthropic-ai/claude-code). Also merge a fallback env PATH '/opt/homebrew/bin:/usr/local/bin:$HOME/.local/bin:/usr/bin:/bin:/usr/sbin:/sbin'. Use Pipe + FileHandle.readabilityHandler for streaming stdout; run waitUntilExit off the main thread. A sandboxed app cannot usefully exec arbitrary user binaries (children inherit the sandbox) — another reason to ship non-sandboxed.
- Sparkle auto-updates: sparkle-project/Sparkle via SPM (https://github.com/sparkle-project/Sparkle); latest is 2.8.0 (modernized update UI + macOS 26 Tahoe fixes; 2.7 added Apple Archive + new delta format). Setup: run `./bin/generate_keys` once (EdDSA private key stored in login Keychain), put the printed public key in Info.plist as `SUPublicEDKey`, set `SUFeedURL` to your hosted appcast.xml, instantiate `SPUStandardUpdaterController` in the SwiftUI App struct. Release flow: archive the app (zip/dmg/tar.xz), drop in an updates folder, run `generate_appcast` — it writes appcast.xml, EdDSA signatures, and delta updates. Host feed + archives on GitHub Pages/Releases (must be HTTPS/modern TLS). Non-sandboxed apps need no XPC services; sandboxed apps require Installer.xpc (`SUEnableInstallerLauncherService`=YES) and optionally Downloader.xpc (`SUEnableDownloaderService`) — and never sign with --deep. Sparkle is prohibited in App Store builds.
- Deployment target: everything this utility needs exists at macOS 14 — MenuBarExtra (13+), .menuBarExtraStyle (13+), SMAppService (13+), Observation/@Observable (14+), SettingsLink (14+). macOS 15-only APIs are window-niceties: windowStyle(.plain), WindowDragGesture, pushWindow, defaultWindowPlacement, defaultLaunchBehavior — irrelevant for a MenuBarExtra .window popover. As of mid-2026 the current OS is macOS 26 Tahoe and Sonoma (14) is in its final security-update year, so min target 14.0 covers Sonoma+Sequoia+Tahoe users; pick 15.0 only if you want the new window-placement APIs or plain-style floating panels.
- Xcode project vs SwiftPM executable: a SwiftPM executableTarget can host a @main SwiftUI app (Swift 5.5+) but yields a bare Mach-O, not a .app bundle — no Info.plist (needed for LSUIElement, SUFeedURL, SUPublicEDKey), no asset catalog/app icon, no entitlements/Hardened Runtime, broken menu bar/activation behavior, and no sane signing/notarization/Sparkle path. Community workarounds exist (swift-bundler https://github.com/stackotter/swift-bundler, XcodeGen, Tuist) but for a shipped, signed, auto-updating app the plain .xcodeproj is the pragmatic choice; keep testable logic in a local SPM package and keep the app target thin.
- Recommended concrete stack: Xcode (16/26) macOS App template, SwiftUI lifecycle, min target macOS 14.0; MenuBarExtra(.window) + LSUIElement=YES + Settings scene; SMAppService.mainApp toggle; App Sandbox OFF, Hardened Runtime ON; FlyingFox (or NWListener) bound to 127.0.0.1 on a configurable port; Process + /bin/zsh -l -c with cached claude path; Sparkle 2.8 via SPM, appcast on GitHub Pages, archives on GitHub Releases; $99 Developer Program, Developer ID Application cert, notarytool submit --wait + stapler; distribute a notarized, stapled .dmg (create-dmg npm tool or hdiutil) plus a ditto zip for Sparkle updates; optional GitHub Actions CI running xcodebuild archive → export → notarytool → stapler → generate_appcast.

## Recommendations
- Pay the $99/yr Apple Developer Program fee before first public release — on macOS 15+ the unsigned-app experience (malware scare dialog + System Settings dance) will lose most non-technical users; budget one afternoon for the Developer ID + notarytool + stapler pipeline and script it (it is fully CLI-automatable).
- Ship non-sandboxed (App Sandbox OFF, Hardened Runtime ON): this is fully supported for Developer ID/notarized direct distribution, removes the network.server entitlement dance for the localhost HTTP server, avoids Sparkle's XPC-service setup, and is the only practical way to exec the user's claude binary. Do not plan for App Store distribution of this app — subprocessing an external CLI is incompatible with the mandatory App Store sandbox.
- Resolve the claude binary once at startup via `/bin/zsh -l -c 'command -v claude'`, fall back to probing ~/.local/bin/claude, /opt/homebrew/bin/claude, /usr/local/bin/claude, cache the absolute path, and surface a clear settings field to override it — this is more reliable than running every invocation through a login shell and avoids ~/.zshrc-only PATH edge cases.
- Set min deployment target to macOS 14.0 (gets @Observable and SettingsLink, covers all supported macOS versions in 2026); do not raise to 15 unless you adopt windowStyle(.plain)/defaultWindowPlacement for custom floating panels.
- Use a plain Xcode project for the app target (Info.plist keys: LSUIElement, SUFeedURL, SUPublicEDKey; entitlements; asset catalog icon) and, if you want SwiftPM ergonomics, move core logic into a local Swift package the app target depends on; skip swift-bundler/XcodeGen for a one-app solo project.
- Adopt Sparkle 2.8.0 via SPM from day one (retrofitting update infrastructure is painful): generate EdDSA keys immediately, keep the private key safe in Keychain (back it up), host appcast.xml on GitHub Pages and update archives on GitHub Releases, and automate generate_appcast in your release script.
- Distribute a notarized+stapled .dmg as the primary download (drag-to-Applications UX) and a ditto-created .zip of the stapled .app as the Sparkle update artifact; notarize both.
